# 2026-05-14 — Building the Agent Company Factory MVP

## Context

One day to convert a content-factory skeleton into an architectural MVP for a multi-agent "company factory". Goal: cover all 6 acceptance criteria from the brief, with tests, docs, and a working UI.

## What worked

- **Boundary-first design.** Defining `Repository<T>`, `Connector`, `LlmProvider`, and `TaskQueue` interfaces before any implementation made the in-memory MVP and the V1 swap obvious in the same breath.
- **Approval as a first-class entity** ([ADR-002](../decisions/ADR-002-domain-model.md)). Pulled "is this approved?" out of every handler, tested it once.
- **Hardening tests as the architecture spec** ([ADR-003](../decisions/ADR-003-orchestrator-hardening.md)). The 12 integration tests in `tests/integration/orchestrator.test.ts` literally are the orchestrator's contract.
- **Mock LLM with deterministic seed.** Tests run in 4s; demo runs in 30s; no API key needed for review.
- **Single aggregator endpoint** (`GET /api/companies/:id` → `CompanyState`). Keeps the UI dumb and SWR happy.

## What didn't

- First pass at the orchestrator skipped the `lockedBy` CAS and relied on JS single-threaded execution. That's correct in production with one Node process but a) hides the V1 design, b) breaks if we ever spawn workers. Reworked to explicit lease.
- First "concurrent claim" test ([BUG-002](../bugs/BUG-002-orchestrator-test-claim-race.md)) accidentally used a task with unmet deps and silently passed/failed depending on timing. Lesson: minimal-repro tasks for concurrency tests.
- `nowIso()` collisions ([BUG-001](../bugs/BUG-001-audit-timestamp-collision.md)) bit us in audit ordering. Sub-millisecond counter fixed it; in V1 we'll use a monotonic DB sequence.

## Decisions

- ADR-001..005 captured.
- 12 explicit hardening guards, each named, tested, and audit-emitting.
- Tailwind v4 + small home-grown shadcn-ish primitives instead of full shadcn init — saved 20 minutes of CLI prompts and gave a coherent dark theme.
